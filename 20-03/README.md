## Домашнее задание к занятию «Микросервисы: подходы» FOPS-38 (Щербатых А.Е.)

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры. Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

### Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

### Ответ 1

Предлагаю единую решение на основе GitLab CI/CD. Эта платформа изначально спроектирована для микросервисной архитектуры и включает все необходимые компоненты из коробки, закрывая ваши требования к облачной разработке, хранению кода, автоматизации сборок и безопасному управлению секретами.

Предлагаемое решение: GitLab CI/CD

GitLab — это единая DevOps-платформа, которая объединяет в себе все необходимые инструменты. Такой подход идеально подходит для микросервисов, так как устраняет сложности с интеграцией разнородных систем и предоставляет единую точку управления.

| Требование	| Реализация в GitLab CI/CD |
|-------------|---------------------------|
| **Облачная система** |	GitLab.com — облачная SaaS-платформа. Можно также развернуть самоуправляемую инсталляцию в облаке (например, в Yandex Cloud) |
| Система контроля версий Git |	GitLab — это полнофункциональный Git-репозиторий с поддержкой распределённой работы |
| Репозиторий на каждый сервис |	Каждый микросервис помещается в свой собственный проект (репозиторий), что обеспечивает изоляцию кода и процессов |
| Запуск сборки по событию из Git |	Триггеры по событиям: push, merge_request, tag и другим |
| Запуск сборки по кнопке с указанием параметров	| Поддерживается ручной запуск пайплайна с возможностью указания входных параметров (when: manual + inputs) |
| Возможность привязать настройки к каждой сборке	| Параметры сборки (переменные, правила запуска) определяются в файле .gitlab-ci.yml в корне репозитория и version control'ятся.
| Возможность создания шаблонов для различных конфигураций сборок |	Механизм include позволяет выносить общие части пайплайнов в отдельные файлы и переиспользовать их |
| Возможность безопасного хранения секретных данных |	Встроенное зашифрованное хранилище Protected Variables (переменные окружения). Можно интегрировать с HashiCorp Vault для централизованного и динамического управления секретами |
| Несколько конфигураций сборки из одного репозитория |	Несколько файлов .gitlab-ci.yml (через include: local) или использование rules для выбора нужного пайплайна в зависимости от ветки, тега или пути |
| Кастомные шаги при сборке	| Любые произвольные команды (bash, python, etc.) в секции script джобы |
| Собственные Docker-образы для сборки |	Поддержка Docker-in-Docker (dind), возможность использовать services для запуска контейнеров во время сборки |
| Возможность развернуть агентов сборки на собственных серверах |	Собственные раннеры (self-hosted runners) на любой инфраструктуре (виртуальные машины, Kubernetes) |
| Возможность параллельного запуска нескольких сборок	| Параллелизм на уровне джобов в одном пайплайне (parallel) и одновременный запуск пайплайнов для разных сервисов.
| Возможность параллельного запуска тестов |	Тесты можно распределять по нескольким джобам, а также использовать parallel: matrix для запуска одного джоба с разными параметрами |

**Обоснование выбора**

GitLab CI/CD идеально подходит для микросервисной архитектуры по следующим причинам:

**Гибкость раннеров**: возможность развёртывания GitLab Runner на собственных серверах или внутри Kubernetes-кластера, что обеспечивает полный контроль над средой сборки и позволяет использовать кастомные Docker-образы с необходимыми зависимостями.

**Масштабируемость**: Система поддерживает параллельное выполнение множества сборок и тестов, динамически масштабируя количество активных агентов в зависимости от текущей нагрузки.

**Управление сложностью**: Благодаря поддержке шаблонов (includes) и иерархических конфигураций, можно унифицировать процесс CI/CD для десятков микросервисов, сохранив при этом возможность специфичной настройки каждого из них.

**Безопасность**: Встроенные механизмы управления переменными, интеграция с внешними хранилищами секретов (Vault) и развитые DevSecOps-функции (SAST/DAST) позволяют поддерживать высокий уровень безопасности на всех этапах жизненного цикла кода.

Такой стек обеспечивает прозрачность процессов разработки, позволяет легко масштабировать инфраструктуру и предоставляет разработчикам удобный инструментарий для быстрой доставки изменений в прод.

### Задача 2: Логи
Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре. Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

  Обоснуйте свой выбор.

### Ответ 2

Для обеспечения централизованного сбора и анализа логов в микросервисной архитектуре по моему мнению лучше использовать стек *Grafana Loki*. Так как он обеспечивает высокую эффективность при минимальных требованиях к ресурсам.

| Компонент |	Роль |
|-----------|------|
| Агент (Promtail / Fluent Bit) |	Собирает логи из stdout контейнеров на каждом узле и пересылает их |
| Хранилище (Loki) |	Централизованное хранилище, индексирующее только метаданные (лейблы) логов |
| Интерфейс (Grafana) |	Единая точка доступа для разработчиков: поиск, визуализация и алертинг |

**Обоснование выбора**

**Минимальные требования к приложениям**: Приложения выводят логи в stdout (стандарт POSIX), что не требует внедрения сторонних библиотек. Агент сбора (например, Promtail или Fluent Bit) работает как «sidecar» или на уровне узла (DaemonSet в K8s), собирая потоки данных прозрачно для разработчиков.

**Эффективность хранения**: В отличие от тяжеловесного Elasticsearch, Loki индексирует только метки (metadata labels), а не всё содержимое лога. Это радикально снижает потребление памяти и дискового пространства, позволяя экономично хранить большие объемы данных.

**Поиск и доступ**: Grafana предоставляет мощный интерфейс (Explore), где разработчики могут быстро фильтровать логи по нужным тегам и выполнять полнотекстовый поиск с помощью языка LogQL.

**Шаринги и сохраненные запросы**: В Grafana реализована возможность сохранения дашбордов и поисковых запросов. Ссылка на результат поиска — это просто URL текущей страницы, которым удобно делиться с коллегами при отладке.

Для задач, где критически важен глубокий полнотекстовый поиск по очень большим массивам неструктурированных данных (и бюджет позволяет содержать тяжелую инфраструктуру), можно рассмотреть Elastic Stack (ELK). Однако, для большинства микросервисных систем Grafana Loki является современным и более сбалансированным стандартом.

### Задача 3: Мониторинг
Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре. Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

Обоснуйте свой выбор.

### Ответ 3

Для обеспечения комплексного мониторинга микросервисной архитектуры рекомендуется использовать стандарт де-факто: стек *Prometheus + Grafana*. Данный стек позволяет полностью закрыть требования к сбору, хранению и визуализации метрик.

| Компонент |	Роль |
|-----------|------|
| Prometheus	| Централизованная система сбора (pull-модель) и временных рядов (TSDB) |
| Node Exporter	| Агент на каждом хосте для сбора инфраструктурных метрик (CPU, RAM, HDD, Network) |
| Exporter'ы сервисов |	Кастомные эндпоинты в сервисах (или sidecar-контейнеры) для специфичных бизнес-метрик |
| Grafana |	Визуализация данных, агрегированные панели и запросы к Prometheus |

**Обоснование выбора**

**Полнота данных**: Использование Node Exporter обеспечивает детализированный мониторинг состояния всех хостов, а прикладные экспортеры позволяют собирать как общие метрики (потребление ресурсов каждым микросервисом), так и специфичные бизнес-показатели.

**Гибкость визуализации**: Grafana обладает широчайшими возможностями по созданию дашбордов. Она позволяет агрегировать данные из множества источников, создавать сложные запросы (через язык PromQL), настраивать переменные для динамических фильтров и делиться готовыми панелями с командой.

**Масштабируемость и надежность**: Prometheus спроектирован для работы в динамических средах (Kubernetes, облака), поддерживает автоматическое обнаружение целей (service discovery) и обеспечивает надежное хранение временных рядов.

**Единая экосистема**: Сочетание Prometheus и Grafana позволяет построить систему «наблюдаемости» (observability), где легко проследить корреляцию между всплеском нагрузки на хост (метрики) и ошибками в приложении (логи Loki) через сквозные ID запросов.

Такой стек является сейчас т.н. "индустриальным стандартом", что гарантирует наличие готовых шаблонов дашбордов, широкую документацию и поддержку сообщества для любых типов сервисов.
